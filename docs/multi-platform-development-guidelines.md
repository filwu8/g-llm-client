# G-LLM Multi-Platform Development Guidelines

这份规范用于判断一个改动应该在 Windows、macOS、Linux 三端共享，还是应该作为平台特性隔离。默认原则是：产品体验和业务逻辑共享，操作系统集成隔离。

## 核心原则

- 默认共享：除非改动依赖某个操作系统能力，否则应进入共享代码，三端一起生效。
- 明确隔离：平台差异必须有清楚理由，例如托盘、Dock、窗口行为、安装器、系统截图、签名、公证、路径约定。
- 不做隐式分叉：不能因为在 Windows 机器上开发，就只让 Windows 包拥有新的前端体验。
- 构建可追溯：正式安装包必须来自已提交的 git commit，不使用未提交本地改动构建发布包。
- 体验优先一致：同一个页面、弹窗、模型列表、设置项在三端应保持同样的信息结构和视觉密度。

## 应该多端同步的改动

这些改动属于共享产品体验，不应该做平台版本隔离。

- 主界面布局、聊天窗口、输入框、工具栏、消息渲染。
- 模型选择、模型排序、模型能力标签、供应商设置、全局默认模型。
- 助手列表、自定义助手、新增助手、助手设置、助手模型配置。
- 空间中心、知识库、记忆、联网搜索、文件和图片附件的产品流程。
- Markdown 渲染、代码块、Mermaid、图片显示、复制文本等通用渲染行为。
- 供应商协议、OpenAI-compatible API 参数、模型能力推断、数据结构和类型定义。
- 本地数据格式、迁移逻辑、导入导出格式。
- 错误提示文案、空状态、加载态、禁用态、搜索和筛选规则。
- CSS 视觉样式、响应式布局、弹窗紧凑度、列表密度。

典型例子：

- 聊天输入区模型下拉显示模型能力标签，应该三端同步。
- 全局默认模型和会话模型使用同一套模型选择器，应该三端同步。
- GPT、Claude、Gemini 等模型排序规则，应该三端同步。
- 自定义助手、助手设置、空间中心顶部更紧凑，应该三端同步。

## 应该平台隔离的改动

这些改动依赖操作系统能力、安装机制或桌面习惯，应隔离实现。

- Windows 托盘、任务栏、悬浮 logo、快速对话最小化行为。
- macOS Dock、菜单栏、关闭窗口习惯、签名和 notarization。
- Linux AppImage、deb、桌面入口、不同桌面环境兼容。
- 安装器和卸载器逻辑，例如 NSIS、DMG、deb postinst。
- 系统截图、系统剪贴板、系统通知、开机启动、文件关联。
- 窗口透明、无边框、置顶、焦点恢复等原生窗口行为。
- 原生依赖打包和动态库处理。
- 平台路径差异，例如 `%APPDATA%`、`~/Library/Application Support`、XDG 目录。
- 代码签名、证书、权限声明、系统安全提示。

典型例子：

- Windows 快速对话按钮显示“最小化”，macOS 显示“关闭”，可以隔离。
- Windows 卸载器增加“是否删除本地数据”页面，只属于 NSIS，不同步到 macOS。
- macOS Gatekeeper、公证提示，不应该影响 Windows/Linux 逻辑。

## 目录和代码边界

- `src/renderer/`：默认视为共享 UI。不要在这里复制一套 Windows UI 和一套 macOS UI。少数平台差异必须很小、可读、并有注释说明原因。
- `src/shared/`：共享类型、模型能力、供应商模板、助手预设、业务规则。不得放平台专用逻辑。
- `src/main/`：主进程和原生能力。平台差异优先放在这里，通过清晰函数暴露给 preload/renderer。
- `src/preload/`：桥接层。向 renderer 暴露能力时，优先暴露“能力”而不是让 UI 到处判断平台。
- `build/`：安装器、打包脚本、平台安装行为。这里的改动通常需要平台验收。
- `docs/`：涉及发布、平台差异、QA 流程的改动必须补文档。

## 判断流程

做改动前先问 5 个问题：

1. 用户是否期望三端看到同样的产品体验？
2. 这个改动是否只依赖 React/CSS/TypeScript 业务逻辑？
3. 是否存在操作系统 API、安装器、签名、窗口管理或路径差异？
4. 如果只在一个平台生效，另两个平台是否会显得落后或不一致？
5. 是否可以通过共享 UI 加平台能力适配完成，而不是复制页面？

判断结果：

- 第 1、2、4 个问题答案为“是”：优先做共享改动。
- 第 3 个问题答案为“是”：只隔离依赖操作系统的最小部分。
- 第 5 个问题答案为“是”：共享 UI，平台能力放 main/preload 适配层。

## 推荐实现方式

共享 UI 调用能力，不直接散落平台判断：

```ts
const canUseScreenshot = window.gllm.capabilities.screenshot
```

平台能力在主进程或 preload 中适配：

```ts
if (process.platform === 'win32') {
  // Windows-specific screenshot/window behavior
}
```

如果当前还没有能力字段，可以暂时使用 `window.gllm.platform`，但要限制范围：

- 可以用于按钮图标、窗口关闭/最小化等很小的行为差异。
- 不应用于复制整页布局、复制整套组件、隐藏共享功能。

## 禁止事项

- 禁止用未提交本地改动构建正式发布包。
- 禁止只在某个平台的本地工作区修改共享 UI，然后直接打包发布。
- 禁止为同一个通用页面维护两套平台组件。
- 禁止把普通 CSS 视觉优化写进平台分支。
- 禁止在安装器脚本中夹带会影响共享数据格式的业务改动。
- 禁止在没有文档和 QA 项的情况下加入新的平台特性。

## 开发工作流

1. 开发前先标记改动类型：`shared`、`platform-windows`、`platform-macos`、`platform-linux`。
2. 共享改动优先修改 `src/renderer/` 和 `src/shared/`。
3. 平台差异只隔离最小实现，例如主进程窗口行为或安装器脚本。
4. 修改后至少执行 `pnpm build`。
5. 涉及平台能力时，在对应系统上跑平台 QA。
6. 发布前执行 `git status -sb`，确认没有未提交代码参与正式构建。
7. 安装包、GitHub Release、下载页应记录或对应同一个 commit/tag。

## Review Checklist

提交或合并前检查：

- [ ] 这个改动已标记为共享或平台特性。
- [ ] 通用 UI 没有被平台分叉。
- [ ] 平台判断只出现在必要的最小范围。
- [ ] 共享数据结构没有放进安装器或平台脚本里修改。
- [ ] 三端应一致的页面在同一套组件/CSS 中实现。
- [ ] 已执行 `pnpm build`。
- [ ] 平台特性已在对应系统验收，或在 PR/提交说明中标明未验收风险。
- [ ] 正式发布包来自干净工作区中的已提交 commit。

## Release Checklist Supplement

每次发版前额外确认：

- `git log -1 --date=iso-local --pretty='%h %ad %s'` 记录到发布说明或内部记录。
- `git status -sb` 必须干净，除非明确是本地测试包。
- Windows、macOS、Linux 的共享 UI 改动来自同一个 tag。
- 如果某个平台使用手工生成的安装包，必须确认它对应的源码 commit。
- 如果下载页更新了安装包链接，必须同时确认版本号、文件名、SHA256 或发布时间。
